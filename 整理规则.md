# 《记录的地平线》设定整理规则

## 1. 目标
- 从 `ログ・ホライズン_小説家になろう.pdf` 中整理世界观、战斗系统、角色、场景、组织及其他相关设定。
- 整理结果服务于“中原（中国服务器）同人创作”，因此必须区分：
  - 可继承到中原服务器的共通设定
  - 明显依赖日本服务器的人、事、地、组织、制度和语境

## 2. 唯一术语标准
- `《记录的地平线》可信译名表.xlsx` 是本项目的唯一术语标准来源。
- 只要 `xlsx` 中已收录某术语，整理时必须优先采用其中已有的可信中文译名。
- 不允许在 `xlsx` 已收录的情况下自行另起译名。
- 若 `xlsx` 中存在多个可能译法，整理结果中应优先采用备注或上下文更明确的版本；如仍无法判断，标记为“待核对”，不得擅自定稿。

## 3. PDF 与 xlsx 的分工
- `pdf` 用于提取设定内容、上下文、叙述逻辑和原文出处。
- `xlsx` 用于统一中文译名、注音信息、版本出处和已有备注判断。
- 每条整理结果都应尽可能同时保留：
  - `PDF原文出处`
  - `xlsx译名依据`

## 4. 每条设定的最少字段
- `设定ID`
- `设定名`
- `日文名`
- `分类`
- `适用范围`
- `对中原服的可用性`
- `摘要`
- `日本服依赖度`
- `关联设定`
- `PDF原文出处`
- `xlsx译名依据`
- `备注`

## 5. 字段说明

### 5.1 分类
- 可使用但不限于以下分类：
  - 世界观
  - 战斗系统
  - 职业与技能
  - 种族
  - 场景与地理
  - 组织与制度
  - 人物
  - 道具与装备
  - 怪物与生物
  - 事件与副本
  - 社会生活
  - 其他设定

### 5.2 适用范围
- 统一使用以下标签之一：
  - `全服共通`
  - `大概率共通`
  - `特定服务器特有`
  - `特定服务器实现形式，底层规则可能共通`
  - `日本服特有`
  - `日本服实现形式，底层规则可能共通`
  - `待考证`

### 5.3 对中原服的可用性
- 统一使用以下标签之一：
  - `可直接沿用`
  - `可改造沿用`
  - `仅作参考`
  - `不建议沿用`
  - `待判断`

### 5.4 日本服依赖度
- 统一使用以下标签之一：
  - `低`
  - `中`
  - `高`

## 6. 原文出处记录规则
- 每条设定都必须尽量提供可回查的出处信息。
- 原则上至少记录以下 4 项：
  - `PDF页码`
  - `章节标题`
  - `关键词`
  - `日文短摘录`
- 由于 PDF 为日文竖排转文本，抽取后可能出现单字换行，因此原文定位不能只依赖一句完整长引文。
- `日文短摘录` 只保留足以定位的短句或关键词组合，不追求长段复制。

## 7. xlsx 译名依据记录规则
- 只要该设定涉及已收录术语，必须记录：
  - `工作表名`
  - `日文原词`
  - `中文译名`
- 如 `xlsx` 中没有对应术语，写明：
  - `xlsx未收录，正文保留原文`
- 如 `xlsx` 中没有对应术语，且该术语为全汉字词，可按项目需要直接写成简体中文汉字，但必须在 `xlsx译名依据` 中注明：
  - `xlsx未收录；本项目按全汉字词直接使用简体中文写法`
- 对于角色称呼、组织简称、技能名、专有名词、地名，必须优先核对 `xlsx`。

## 8. 整理时的判断原则

### 8.1 先抽设定，再定译名
- 先从 `pdf` 识别出设定条目及其上下文，再回查 `xlsx` 统一中文译名。
- 不允许因为先入为主的中文习惯而跳过 `xlsx` 核对。

### 8.2 共通设定优先
- 对中原服创作最有价值的内容优先整理：
  - 世界运行规则
  - 玩家/冒险者机制
  - 战斗系统
  - 职业与种族体系
  - 死亡、复活、通讯、道具、生产、经济等系统规则
  - 大地人、区域、迷宫、城市、服务器与世界结构

### 8.3 日本服专属内容单列
- 以下内容优先标为日本服专属或高依赖：
  - 日本服务器具体角色
  - 日本服务器特定公会、政治结构、治理方案
  - 日本地区专属地名与城镇生态
  - 明显只在日本服情境下成立的社会关系与事件

### 8.4 人物条目谨慎处理
- 日本服人物可整理，但必须与“可迁移设定”严格分开。
- 对人物相关设定，要特别区分：
  - 角色个人属性
  - 角色所属系统规则
  - 角色所处日本服社会环境

### 8.5 归属优先于现象拆分
- 整理词条时，优先将设定归入其所属的“对象、群体、地点、组织、事件”本体词条，而不是把所有现象都拆成独立词条。
- 只有在以下情况下，才考虑将现象单独立条：
  - 该现象会跨越多个对象反复出现
  - 该现象本身构成稳定规则或核心机制
  - 该现象后续会被高频独立引用
- 例如：
  - 冒险者的空腹、睡眠、排泄，应优先写入“冒险者”词条
  - 大地人的生理现象，应优先写入“大地人”词条
  - 某城市市场的变化，应优先写入该城市词条
  - 大灾害导致的普遍性系统变化，应优先写入“大灾害”或对应总体现象词条

## 9. 未收录术语处理
- 若术语在 `xlsx` 中未收录：
  - 先保留日文原词
  - 明确标记 `待核对`
- 不得自行创造新的中文译名。
- 若术语为全汉字词，可按项目约定直接使用简体中文汉字写法，但这属于字形转换，不属于新译名。
- 若术语含假名、拉丁字母或混合写法，原则上统一保留原文。

## 10. 输出风格要求
- 以资料整理为主，不写成散文式总结。
- 摘要应准确、克制，尽量避免二次发挥。
- 推断性内容必须显式标注为“推断”或“待考证”，不得与原文明示内容混写。
- 对中原服适配建议可以写，但必须与原作原设定分开。
- 条目标题统一写作：
  - 若已有可信中文译名：`编号 + 中文名 + 日文名`
  - 若 `xlsx` 未收录且不宜自拟：`编号 + 日文原名`
  - 例如：`## GEO-003 妖精环（妖精の輪）`
  - 例如：`## JPN-ORG-010 Ｐｌａｎｔ　ｈｗｙａｄｅｎ`

## 10.1 文件名与发布兼容性规则
- 词条正文的标题显示名，可以保留原作写法、全角字符、日文假名、片假名、拉丁字母原文等。
- 但词条文件名必须优先考虑 GitHub、静态服务器、浏览器上传与跨平台文件系统的兼容性，不能单纯照搬标题显示名。
- 文件名应尽量使用稳定的 ASCII slug，推荐结构为：
  - `编号_英文或拼写化slug.md`
  - 例如：`JPN-ORG-010_Plant_hwyaden.md`
- 文件名中应尽量避免以下高风险形式：
  - 全角拉丁字母
  - 全角空格
  - 不必要的混合空白字符
  - 可能导致 URL 编码、浏览器上传或跨平台同步不稳定的特殊字符
- 若术语本身 `xlsx` 未收录，正文标题仍保留原文，不因为文件名改成 ASCII slug 就反向改动条目标题或正文术语。
- 也就是说：
  - `条目标题` 负责忠实显示原文
  - `文件名` 负责稳定存储、上传与跳转

## 10.2 文件名修改时的联动规则
- 若某个词条文件名因发布兼容性需要调整，必须同步检查并修正所有引用位置。
- 至少要检查：
  - `词条索引.md`
  - `index.html`
  - 其他词条中的 Markdown 跳转链接
  - 任何静态阅读入口中的 `reader.html?file=...`
- 若只是文件名改动，而正文术语未改，则应只修正跳转目标，不要擅自改写条目标题和正文中的术语写法。

## 11. 设定随时间变化的记录规则
- 同一项设定可能会随着剧情推进、版本更新、世界变化或角色认知变化而出现不同阶段的状态。
- 整理时不得以后期信息覆盖前期信息，也不得把前期观察误写成永恒规则。
- 遇到同一设定在不同时点发生变化时，应同时保留：
  - `早期状态`
  - `后期变化`
  - `变化触发条件 / 相关事件 / 相关道具（若已明确）`
  - `各自出处`

### 11.1 需要特别注意的变化类型
- 系统规则本身发生改变
- 角色或群体的外貌、能力、社会地位随时间变化
- 前期只是“主角视角下的认知”，后期获得了更完整或被修正的信息
- 原本存在普遍规则，后期出现例外、破解手段或特殊道具

### 11.2 整理写法要求
- 对同一设定，应优先使用如下结构之一：
  - `初期状态 / 后续变化 / 备注`
  - `常规规则 / 后续例外 / 备注`
  - `主角早期认知 / 后续确认情况 / 备注`
- 例如：
  - `初期状态：冒险者通常会复苏。`
  - `后续变化：后文出现可导致冒险者真正死亡的特殊条件或道具。`
  - `备注：两者并存记录，不以后者覆盖前者。`

### 11.3 判断原则
- 若前期内容只是人物猜测、局部观察或阶段性现象，应明确标注其时点与视角。
- 若后期内容构成补充、修正或例外，也应明确标注“这是后续追加信息”，不得伪装成一开始就已成立的完整规则。

## 12. 自然语言战斗描写判定规则
- 小说中的战斗和状态变化常通过自然语言、角色体感、环境描写和主观视角表达，不能直接把单句文学描写等同于系统规则。
- 凡涉及以下内容，必须进行判定后再整理：
  - 受击后的身体感受
  - 行动迟滞、沉重、麻痹、眩晕等表现
  - 视野、听觉、平衡感异常
  - 疼痛、恐惧、压迫感、疲劳感
  - 场地、天气、地形与技能效果可能混淆的场景

### 11.1 判定等级
- 统一使用以下标签之一：
  - `明示设定`
  - `高概率效果`
  - `情境性表现`
  - `主观体感`
  - `待考证`

### 11.2 各等级含义
- `明示设定`
  - 原文明确说明了技能、状态、系统或攻击效果。
  - 例如直接写出附加效果、异常状态、行动限制、数值或机制说明。
- `高概率效果`
  - 原文虽未规则化定义，但同类效果在多个场景中反复出现，且与同一技能、状态或机制稳定绑定。
- `情境性表现`
  - 只能确认此现象发生在该次战斗或该环境中，不能上升为普遍规则。
  - 例如泥地、雨天、狭窄地形、负重等因素可能共同造成结果。
- `主观体感`
  - 只能确认角色如此感受到，不能直接判定为系统效果。
  - 例如“腿部很重”“身体像灌了铅”“视野发黑”等。
- `待考证`
  - 目前无法判断是技能效果、环境因素、心理作用还是文学修辞。

### 11.3 判定时必须检查的要点
- 是否存在明确的技能名、攻击名、状态名或施术者说明。
- 该句是旁白客观陈述，还是角色主观感受。
- 当时是否存在足以造成类似结果的环境因素。
- 后续是否出现可验证的行为后果。
  - 例如移动下降、站立困难、闪避失败、队友点明异常状态等。
- 同类表现是否在其他场景中重复出现。

### 11.4 整理写法要求
- 证据不足时，不得把自然语言描写直接改写成确定的系统规则。
- 应优先写成：
  - `场景表现`
  - `判定等级`
  - `可能解释`
  - `备注`
- 例如：
  - `场景表现：角色受击后感到腿部沉重。`
  - `判定等级：待考证。`
  - `可能解释：技能附加减速 / 地形拖滞 / 主观体感。`
  - `备注：原文未明示其为固定系统效果。`

## 13. 工作流程
- 每一轮整理都按以下顺序执行：
  1. 读取本规则文件
  2. 读取 `整理进度.md`
  3. 从 `pdf` 抽取当前批次内容
  4. 识别术语并逐项核对 `xlsx`
  5. 检查当前条目是否存在“前期状态 / 后续变化 / 例外条件”
  6. 生成带出处的设定条目
  7. 记录已完成范围、待核对项和术语缺口到 `整理进度.md`

## 14. 顺序整理 + 词条回填整编
- 本项目采用“顺序整理 + 词条回填整编”双段式方法。
- `顺序整理`：
  - 仍以 PDF 原有阅读顺序为主，先把新出现的设定、人物、地点、组织和变化线索摘出。
  - 这一阶段的目标是保住信息和出处，不追求条目结构已经完整。
- `词条回填整编`：
  - 当某个词条已经具备独立成条价值时，不能只使用当前章节材料，必须主动回查该词条在前文和后文的可确认信息。
  - 回填时优先补足：
    - `初次确认 / 基本定义`
    - `成立或出现时点`
    - `成员、构成、领导者（如适用）`
    - `功能、制度、地位`
    - `前期状态 / 后续变化 / 例外`
  - 不得只因为某条信息在后期戏份更重，就让整个词条只剩后期内容。
- 词条整编后的目标风格应接近百科页，而不是流水账摘录页。

### 14.1 词条结构优先级
- 组织词条优先补：
  - `摘要`
  - `成立背景`
  - `成立 / 出现时点`
  - `成员构成 / 领导者`
  - `功能或制度位置`
  - `阶段变化`
- 地点词条优先补：
  - `基本定义`
  - `地理或服务器位置`
  - `所属势力或邻接政体`
  - `用途 / 机能`
  - `阶段变化`
- 系统词条优先补：
  - `定义`
  - `触发条件`
  - `限制`
  - `例外`
  - `后续变化`
- 人物词条优先补：
  - `身份`
  - `所属组织或地域`
  - `在设定上的承载作用`
  - `阶段变化`
  - `相关称谓`

### 14.2 整编时的证据约束
- 回填整编时，仍必须遵守：
  - 所有新增事实都要附 `PDF原文出处`
  - 所有已收录术语都必须回查 `xlsx`
  - 不能凭记忆补完整成员表、完整制度表或完整时间线
- 若只能确认“已直接出现并可查证的部分”，就明确写成：
  - `当前已确认`
  - `当前材料可直接确认`
  - `完整情况仍需后文补证`

## 15. 禁止事项
- 禁止忽略 `xlsx` 已有译名而自行翻译。
- 禁止输出没有出处锚点的“印象流设定”。
- 禁止把日本服局部经验直接写成全服通则。
- 禁止把推测内容伪装成原文事实。
- 禁止以后期设定直接覆盖前期设定而不保留变化过程。
